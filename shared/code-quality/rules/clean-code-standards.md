# Clean Code & Maintainability Standards

## Description
Enforces strict Clean Code guidelines across all code generated by AI agents and developers. Ensures that code is self-documenting, readable for fresher/junior developers, and scalable over long-term project lifecycles.

## Constraints

### 1. Mandatory TSDoc / JSDoc Block Comments
- Every exported class, interface, service method, controller endpoint, and utility function MUST include TSDoc block comments (`/** ... */`).
- Block comments MUST document business purpose, `@param` descriptions, `@returns` payload description, and `@throws` exceptions.
- Undocumented public API methods are STRICTLY FORBIDDEN.

### 2. Magic Value Prohibition Rule
- Magic numbers, inline string literals, and un-enumed status flags MUST NOT be used in conditional logic.
- All status flags, roles, and constant values MUST be defined using TypeScript `enum` or `readonly` constant objects.

### 3. Function Length & Parameter Limits
- Functions MUST NOT exceed 35 lines of code without refactoring into helper methods.
- Functions MUST NOT accept more than 3 positional parameters; functions requiring 4 or more inputs MUST use a typed DTO class or options object.

### 4. Guard Clauses / Early Return Requirement
- Code MUST NOT contain nested `if/else` conditions deeper than 2 levels.
- Precondition validation MUST use guard clauses (early `return` or early `throw`) to flatten execution flow.

### 5. Intention-Revealing Naming
- Variable and function names MUST be descriptive and self-explanatory (e.g. `calculateLoanAmortizationSchedule` instead of `calc`).
- Single-letter variable names (e.g. `x`, `temp`, `data2`) are forbidden except in simple loop counter indexes (`i`, `j`).
